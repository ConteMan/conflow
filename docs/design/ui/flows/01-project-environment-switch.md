# 状态矩阵：项目加载与环境切换

## 场景

`conflow serve` 对应一个 workspace（一个项目）。GUI 打开即载入该项目；环境切换是全局动作，影响所有页面的数据视角。

## 状态矩阵

| 状态 | 触发 | UI 表现 | 出口 |
|---|---|---|---|
| 入口 | 打开 GUI / bootstrap | 全局壳先渲染，项目区骨架屏 | 正常 / 加载 / 错误 |
| 加载 | `GET /bootstrap` 进行中 | 顶部项目与环境区骨架屏；不闪烁空文案 | 正常 / 错误 |
| 正常 | 项目载入成功 | 常驻显示项目名 + 当前环境徽标；Production 徽标持续使用危险色描边 | 环境切换 / 各功能页 |
| 空 | 项目无环境（新 init） | 环境位显示「尚未创建环境」+ 创建入口；功能页降级为引导态 | 创建环境 |
| 错误 | workspace 损坏 / 服务不可达 | 全页错误态：原因 + 「重试」+ 指向 CLI `validate` 的修复提示 | 重试 |
| 冲突 | 清单被外部修改（编辑器/Git） | 非阻断横幅「项目文件已在别处修改」+ 「重新载入」；未保存编辑给出保留/放弃选择 | 重载 |
| 危险确认 | 切换到 Production 环境 | 不弹确认框；切换后全局出现持续的 Production 视觉状态（顶栏变调 + 徽标） | — |
| 成功 | 切换完成 | 数据视角整体切换；页面内保留位置与筛选条件 | — |

## 关键决策（待原型验证）

- 切到 Production **不加弹窗**：切换是读操作，减速留给写操作；靠持续视觉状态防误改。这是假设，需在原型评审中确认 PM 不会忽略状态。
- 环境切换时未保存的编辑如何处理：跟随环境（草稿按环境隔离）还是提示丢弃 —— 依赖 Spec 004 草稿分层的结论。

## 文案基调

- 「Production」在 UI 中始终用环境显示名 +「生产」标签，不缩写。
- 外部修改提示不指责来源（「已在别处修改」而非「有人改了」）。
