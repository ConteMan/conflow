# Spec 025：发布计划失效分层与自动重建

> 状态：待实现  
> 依赖：Spec 008（计划语义 diff 与异步 Operation）  
> 关联：Issue #55

## 目标

计划因系统正常节律失效时自动重建、用户无感；只有线上配置被外部修改等异常情况才需要用户知晓并确认，且以页内横幅而非模态弹窗呈现。

## 背景

计划有 15 分钟 TTL，草稿演进或远端变化也会使其失效。当前 UI 把所有失效一视同仁地弹模态"这份发布计划已失效"，要求手动重建；同屏还有顶部横幅、底部横幅、右侧提示共四处重复。重建计划是只读操作（pull → 编译 → diff），对常规失效自动执行没有风险；模态确认应该保留给真正需要人介入的异常。

## 范围

### 服务端：结构化失效原因

计划读模型的 `invalidation_reason`（当前为自由文案）扩展为结构化对象：

```json
{
  "invalidation": {
    "code": "plan_expired",
    "tier": "routine",
    "message": "计划已过期，请重新构建。"
  }
}
```

| code | tier | 场景 |
|---|---|---|
| `plan_expired` | `routine` | TTL 到期 |
| `draft_advanced` | `routine` | 本地草稿 revision 前进（Conflow 内正常编辑） |
| `remote_changed` | `external` | 远端 ETag 变化：线上配置被 Conflow 之外的渠道修改 |
| `pack_changed` | `external` | Pack ref 或 schema 版本变化 |

- 兼容：保留原 `invalidation_reason` 文案字段一个版本周期，新旧字段并存。
- 多重原因同时成立时取最高 tier（`external` 优先）。

### 前端：分层处置

- **routine**：进入计划页或轮询发现失效时，自动触发重建，主区显示"正在重新构建计划…"进行态。防循环保护：一次导航内自动重建至多 1 次；若新计划立即再次失效，停止自动化并按 external 方式提示。
- **external**：不弹模态。页内单一横幅说明具体原因（如"线上配置发生了 Conflow 之外的修改（版本 6 → 7）"），高亮"重新构建计划"按钮，由用户确认。
- 删除现有失效模态弹窗；顶部横幅、底部横幅、右侧提示三处重复收敛为一处横幅。

### API / CLI

- 计划响应结构新增 `invalidation` 对象（OpenAPI 同步）。
- CLI `conflow plan` 输出的 JSON envelope 携带同一结构；CLI 行为不变（本就每次重建）。

## 非范围

- 计划 TTL 时长调整。
- 后台定时自动重建（仅在用户进入计划页时触发）。
- 发布步骤页内的 ETag 冲突处理（既有合同已覆盖）。

## 验收

- TTL 过期后进入计划页：不出现模态，自动重建并在完成后展示新计划。
- 修改草稿后进入计划页：同上自动重建，diff 反映新修改。
- 在 Firebase 控制台手工改一个参数后进入计划页：不自动重建，页内横幅提示外部修改，点击按钮后重建成功。
- 自动重建后的计划立即再次失效时：停止自动重试，显示横幅。
- OpenAPI 合同更新且 `make check` 通过；服务端单测覆盖四种 code 与 tier 判定。
